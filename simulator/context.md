 Two-Tower 지표가 쓰레기로 나오는 이유 — 진단 보고서
                               
  0. 핵심 결론 한 줄           
                                                                                                                                                             
  모델이 학습이 안 된 게 아니라 "카테고리·페르소나 분류기"로 학습됐다. 그래서 retrieval 지표가 random 수준입니다.

  2. 가능한 원인 후보부터 제거

  데이터 결함부터 의심해서 직접 확인했는데 모두 정상:
  - 학습용 positive pair: 1,484,796건 (충분)
  - 모든 test 유저가 train(cart/purchase)에 등장 (cold-user 0%)
  - test 상품 91.8%가 train에 등장 (cold-item 거의 없음)
  - test purchase pair의 89.9%가 (유저+상품) 모두 학습된 조합

  → 데이터 분할/콜드스타트 문제 아님. 모델 설계 문제.

  3. 진짜 원인 — 임베딩을 직접 까보면 명확함

  저장된 두 타워 임베딩을 분석한 결과 (item_embeddings.npy, two_tower.pth):

  3-A. Item 타워: 카테고리에 완전히 흡수됨

  같은 category_L1 안 상품들의 평균 코사인 유사도:

  Baby/Children   n= 4,084   avg cos = 0.854
  Divided         n=11,042   avg cos = 0.905
  Ladieswear      n=27,916   avg cos = 0.887
  Menswear        n= 5,299   avg cos = 0.899
  Sport           n= 1,659   avg cos = 0.941

  같은 대분류 상품 5만 개가 사실상 한 점 근처에 모여 있습니다. Top-300 안에 진짜 정답 1개가 들어갈 가능성이 거의 없죠.

  원인 — phase2_two_tower.py:151-159 item tower 입력 구성:
  [item_emb(32) | cat1(32) | cat2(32) | cat3(32) | tier(32)] = 160차원
  - 입력의 80%가 카테고리/가격대 신호 → 학습이 카테고리 위주로 수렴
  - category_L1은 5개, L2는 ~10개, L3는 ~20개. 조합이 매우 적음
  - Ladieswear 한 카테고리가 56% (27,916/50,000)로 카탈로그 편향까지 강함

  3-B. User 타워: 페르소나로만 클러스터링되고 그것마저 약함

  페르소나 centroid 간 cosine 유사도 행렬:
          p0    p1    p2    p3    p4    p5
  p0    [1.00  0.78  0.95  0.88  0.79  0.84]
  p1    [0.78  1.00  0.76  0.83  0.71  0.81]
  p2    [0.95  0.76  1.00  0.84  0.75  0.84]   ← p0과 0.95
  p3    [0.88  0.83  0.84  1.00  0.73  0.91]
  p4    [0.79  0.71  0.75  0.73  1.00  0.73]
  p5    [0.84  0.81  0.84  0.91  0.73  1.00]

  페르소나끼리도 0.71~0.95라 거의 분리가 안 됩니다. p0와 p2는 0.95로 사실상 같은 방향.

  원인 — phase2_two_tower.py:147-149 user tower 입력:
  [user_emb(32) | persona_emb(32)] = 64차원
  - 입력의 50%가 6개짜리 페르소나 → 학습 신호 대부분이 페르소나로 흘러감
  - 유저별 행동 시퀀스/카테고리 선호 같은 차별 피처 없음
  - in-batch softmax는 강한 신호로 빨리 수렴 → user_idx 임베딩은 거의 학습되지 않음

  3-C. In-batch sampled softmax — 표준 보정 누락

  phase2_two_tower.py:204 의 in-batch softmax는 인기 아이템에 대한 강한 popularity bias가 있습니다 (인기 아이템이 같은 배치에 자주 등장 → over-penalize).
  표준 해결책은 logQ correction (logits -= log(P(item))). 현재 코드에 없음.

  영향: 인기 상품 임베딩이 잘못된 방향으로 밀림 → 다양성·정확도 모두 저하.

  3-D. 보조 요인

  - EPOCHS=5 : retrieval 모델 치고 너무 짧음 (보통 20~50)
  - TEMPERATURE=0.07 : CLIP의 큰 배치/고차원 가정. 64차원·1024 배치엔 0.1~0.2가 더 안전
  - drop_last=True + 1024 batch 자체는 OK이나 위 1)~3)과 결합되면 학습 신호 약화

  ---
  4. 왜 HitRate@50 / NDCG@50이 정확히 0이 나왔나

  - Stage1 (Two-Tower) 후보 300개에 정답이 들어갈 확률이 0.6%
  - 그 0.6% 케이스에서 Stage2 (DeepFM)가 50개로 더 좁힐 때 정답이 살아남을 확률을 곱하면 평가 샘플 500건에선 1건도 안 잡힐 수 있음
  - 즉 "DeepFM이 못한 게 아니라 Stage1이 정답을 후보에 넣지 못함"이 본질
